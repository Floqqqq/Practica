# Отчёт по фронтенд-части проекта «Контекст»

## 1. Исходные данные

Проект выполнен по заданию на учебную практику направления 09.03.04 «Программная инженерия»: разработка интеллектуальной поисковой системы по внутренней базе знаний университета. Роль — frontend-инженер (FE). Рабочая ветка — front/dev.

Цель фронтенд-части: предоставить адаптивный веб-интерфейс для загрузки учебных документов и полнотекстового поиска по найденным фрагментам.

## 2. Выбранные технологии

- React 18 и TypeScript — компонентная архитектура и строгая типизация;
- Vite — разработка и production-сборка;
- XMLHttpRequest — реальный прогресс multipart-загрузки;
- Fetch API и AbortController — поиск и отмена устаревших запросов;
- CSS без внешней UI-библиотеки — адаптивный интерфейс и минимальный размер зависимостей;
- Nginx — раздача production-статики и проксирование API.

## 4. Архитектура

Страница KnowledgeBasePage координирует состояние загрузок, поисковый запрос, историю и пагинацию. Компоненты не знают деталей HTTP-контракта. Нормализация ответов сосредоточена в services/api.ts, а временная клиентская регистрация документов — в services/storage.ts.

Файлы валидируются до отправки: разрешены расширения PDF/DOCX, запрещены пустые файлы и размер более 20 МБ. Ошибка одного элемента не останавливает обработку остальных файлов.

## 5. Интеграция с бэкендом

Фронтенд согласован с реализованным POST /api/v1/documents/upload и его полями id, file_name, size, status, pages_count и duplicate. Множественная загрузка раскладывается на запросы с multipart-полем file.

На момент реализации search_handler.go и swagger.yaml в бэкенде пусты, а роутер регистрирует только upload и health. Поэтому поисковый UI и нормализатор готовы к контракту задания GET /api/v1/search, но сквозной поиск потребует завершения этого эндпоинта бэкенд-командой. Эндпоинт выдачи списка документов также отсутствует; до его появления FE-03 использует localStorage без демонстрационных подмен API.

## 6. Проверка качества

Проведены следующие проверки:

- визуально просмотрены все 7 страниц исходного PDF и сопоставлены FE-01–FE-09;
- синтаксическая диагностика 16 файлов TypeScript/TSX — 0 ошибок;
- локальный запуск интерфейса в браузере — страница и семантическая структура загружаются;
- поиск с тестовым API — 10 карточек на первой странице, 2 на второй;
- подсветка запроса «жизненный цикл» — совпадения выделены;
- пагинация — переход на вторую страницу даёт 2 результата и aria-current=2;
- viewport 320 px — ширина документа соответствует viewport, основные элементы остаются читаемыми;
- проверены пустое состояние, история запроса и адаптивное расположение панелей.

Для окончательной командной приёмки после реализации search API необходимо выполнить npm install, npm run typecheck, npm run build и сквозной E2E-сценарий «загрузка → индексация → поиск → результат» в Docker Compose.
